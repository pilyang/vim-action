## 결정

architecture(현재 구조)·decisions(결정 히스토리)에 더해, 진행 중인 멀티세션 작업의 플랜을 담는 세 번째 스킬 `plans`를 도입한다. 장기 기억인 앞의 둘과 달리 단기 기억으로 — 살아있는 문서로 계속 갱신되고, 완료되면 사용자 확인 후 삭제된다.

## 배경·근거 (왜)

세션이 끝나면 진행 중이던 플랜이 컨텍스트와 함께 사라져, 다음 세션이 git diff나 대화 기억으로 진행 상태를 역추적해야 했다. "지금 무엇을 하는 중인가"를 담는 SSOT가 필요했다. 낡은 플랜의 컨텍스트 주입이 최대 리스크이므로 수명 관리 규칙(활성 최대 3개, 갱신일 강제, 완료 시 삭제, 아카이브 금지)을 스킬의 중심에 두었다. 플랜 문서는 git에 커밋해 머신·세션 간 공유한다. planning method(플랜을 세우는 법)는 다루지 않는다 — plan mode 등 기존 도구와의 역할 충돌을 피하기 위함.

## 검토한 대안

- **decisions 스킬에 통합**: decisions는 불변 append-only가 핵심 불변식인데 플랜은 계속 갱신·삭제되는 정반대 수명이라, 섞으면 둘 다 망가져 기각.
- **스킬 없이 PLAN.md 컨벤션 파일만 사용**: "이어서 하자" 같은 발화에서의 자동 트리거와 갱신·정리 워크플로우 강제가 안 되어 기각.
- **완료 플랜 아카이브 보관**: 히스토리 가치가 있는 내용은 decisions 담당이고, 완료 플랜 무덤은 아무도 읽지 않으면서 낡은 컨텍스트 주입 위험만 남겨 기각. git 히스토리가 복구 수단.
