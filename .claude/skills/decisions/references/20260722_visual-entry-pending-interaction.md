# Visual 진입과 Normal pending의 상호작용 — 특례 없이 기존 문법 규칙 적용

## 결정

Normal에서 pending 누적 중 `v`/`V` 입력에 **새 규칙을 만들지 않고 기존 문법 규칙을 그대로 적용**한다:

- **`3v`** (카운트 누적 중): 카운트를 버리고 Visual 진입 — mode-change 키의 count 무시 원칙(`3i`와 동일).
- **`dv`** (오퍼레이터 대기 중): invalid no-op — `v`는 opMotion 화이트리스트 밖이므로 pending과 키를 함께 버리고 Normal 유지, Visual 진입하지 않음(`dq`와 동일, [20260712 무효 시퀀스 no-op](20260712_pending-invalid-sequence-noop.md)).

둘 다 픽스처로 명문화한다.

## 배경·근거 (왜)

실제 Vim에서 이 조합들은 고유 의미가 있다 — `{count}v`는 count문자 선택 후 진입, `dv`는 d의 동작을 charwise로 강제. 둘 다 v1 어휘 범위 밖이라 "지원하지 않는 조합의 안전한 무력화"가 문제였고, 기존 두 결정이 이미 답을 내놓고 있었다:

- 카운트 무시는 파괴적이지 않고(`3gg`·`3i` 선례), 무효 연속의 no-op은 "파괴적 편집은 오해석을 수용하지 않는다"는 기준과 정합적이다 — `d` 문맥의 `v`에 임의 의미를 부여하지 않는다.
- **미래 호환**: v2에서 진짜 Vim 의미를 지원하게 되면 `dv`는 no-op → 새 의미라 순수 확장(기존 동작 안 깨짐)이고, `3v`는 무시 → 해석이라 오해석보다 교정 비용이 낮다.
- 구현·픽스처·문서 어디에도 예외 분기가 생기지 않는다. v2의 `Ctrl-v`(Visual-block) 등 새 모드 키에도 같은 원칙이 그대로 적용된다.

Vim과 다른 의미론임을 인지한 의도적 이연이다 (`dv` 무반응은 v1에서 Vim 기대를 채울 방법이 없는 상황의 최선).

## 검토한 대안

- **`v`가 항상 이김** (pending 무조건 폐기 + Visual 진입): "오퍼레이터 대기 중 화이트리스트 밖 키 = invalid"라는 단일 규칙에 `v`만 예외가 되고, v2에서 `dv`의 진짜 의미(charwise 강제)를 지원하려 할 때 기존 동작 교체(breaking)가 된다. 기각.

## 영향 범위

- 코드: `VimEngine.swift` — 최상위에 `v`/`V` 추가만으로 기존 문법이 위 동작을 자동 산출(특례 코드 없음). Visual 픽스처에 `3v`·`dv` 케이스 명문화.
- 재검토 조건: v2에서 `{count}v` 또는 오퍼레이터 forced-motion(`dv`/`dV`)을 지원하기로 하면 이 결정을 supersede.
