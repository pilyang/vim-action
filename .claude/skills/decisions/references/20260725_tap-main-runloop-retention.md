# 탭 메인 런루프 유지 확정 — 전용 스레드 전환 기각 (재검토 트리거 명시)

- **결정일**: 2026-07-25

## 결정

CGEventTap run-loop 소스의 **메인 런루프 부착을 유지**한다 — [20260712 결정](20260712_main-runloop-tap-attachment.md)이 예약한 "엔진 연결 전 재검토"의 이행. 전용 CFRunLoop 스레드 전환은 기각하되, 아래 재검토 트리거가 충족되면 재개한다. 유지의 성립 조건인 콜백 경량 불변식은 [20260725_callback-light-invariant.md](20260725_callback-light-invariant.md)로 분리 기록.

## 배경·근거 (왜)

3-에이전트 리뷰(독립 평가·적대적 비판·실코드 검증, 2026-07-25)로 "디스패처 전 선제 전환" 권장안을 검증한 결과 기각이 수렴됐다:

1. **OS 탭 타임아웃(~1초)은 콜백 실행 스레드와 무관하게 이벤트 단위로 발동한다.** 전용 스레드가 고치는 것은 "무관한 메인 스톨이 가벼운 콜백을 굶기는" 경우뿐이고, "콜백 자체가 느린" 경우(디스패처 AX 프로브 리스크의 본체)는 어느 스레드에서든 탭이 꺼진다 — 전환의 핵심 추동력이었던 근거가 문제와 처방이 어긋나 있었다.
2. **"무관한 메인 스톨" 시나리오 자체가 희귀하다.** LSUIElement 메뉴바 앱이라 메인 스레드가 평소 유휴이고, 실측(2026-07-21~25, 4일 실사용)에서 스톨 유래(`tapDisabledBy*`) 비활성화 0건.
3. **실패 모드가 양성 degrade다.** 최악이 "타이핑 ≤1초 지연 → 탭 off → 키 즉시 통과, Vim 기능만 일시 정지"이며, 복구는 워치독+스톨 게이트가 담당(기설계). 키보드 인질 상태는 발생하지 않는다.
4. **전환의 실비용이 플랜 추정보다 크다** (실코드 검증): `assumeIsolated` 제거가 촉발하는 nonisolated 연쇄(엔진 격리·mode 메인 홉), KeyTranslator TIS 프리웜 재구성, 그리고 플랜에 없던 숨은 결합 2개 — SwiftUI 토글 didSet→엔진 재생성의 메인 고정(SSOT 불변식과 충돌), 콜백 최상단 `isInterceptionEnabled` 관찰 프로퍼티 읽기의 원자 미러 필요. 배포 전 1인 프로젝트에서 관측 0건의 양성 degrade를 없애는 대가로 부적절하고, 스레딩 버그는 스톨보다 디버깅이 어렵다.
5. 실측 0건은 "가벼운 콜백" 조건의 산물이므로 유지의 근거는 측정이 아니라 **"콜백을 계속 가볍게 유지할 수 있다"는 설계 논거**([콜백 경량 불변식](20260725_callback-light-invariant.md))에 둔다.

## 검토한 대안

- **전용 CFRunLoop 스레드 선제 전환 (디스패처 전)**: 위 1(타임아웃 스레드 무관)·2(시나리오 희귀)·4(비용·신규 리스크)로 기각. "할 거라면 디스패처 전이 반경이 작다"는 명제 자체는 실코드로 참임을 확인 — 기각 이유는 시점이 아니라 필요성.

## 재검토 트리거

다음 **셋 모두** 충족 시 전환을 재개한다 (하나라도 빠지면 전환은 헤드라인 이득을 못 준다):

1. 디스패처 배포 후 스톨 유래(`tapDisabledBy*`) 탭 비활성화가 실측으로 유의미
2. 콜백 경량 불변식으로 지울 수 없는 잔여 스톨임이 확인됨
3. TIS 오프메인 번역 경로(레이아웃 프리웜 스냅샷) 실증 — 없으면 전환해도 메인 커플링이 재도입됨

별도 트리거: 앱이 메인 스레드 상시 부하(실시간 오버레이 UI 등)를 갖게 되면 1~2와 무관하게 재검토.
